# Tasks: Bulk User Import

## Task List

### T-001: Database Migration
- **Scope:** Create a migration that adds the `import_batch_id` column (UUID, nullable, indexed) to the `users` table.
- **Inputs:** Design document (data model section)
- **Dependencies:** None
- **Expected Output:** Migration file that can be run forwards and rolled back.
- **Definition of Done:** Migration runs successfully, column exists, index is created, rollback works.
- **Validation Notes:** Check that migration is backwards-compatible, column is nullable, index exists.

### T-002: CSV Parser
- **Scope:** Implement a streaming CSV parser that reads a file, validates headers against the expected schema, and extracts rows as structured objects.
- **Inputs:** Design document (CSV parser component), expected column schema
- **Dependencies:** T-001
- **Expected Output:** Parser module that accepts a file stream and returns parsed rows or a structural error.
- **Definition of Done:** Parser handles valid files, rejects structurally invalid files, streams without loading entire file into memory. Unit tests pass.
- **Validation Notes:** Test with empty files, missing headers, malformed CSV, large files.

### T-003: Structural Validator
- **Scope:** Implement validation for CSV structure: required headers present, no empty required fields in rows, correct row format.
- **Inputs:** Parsed CSV data from T-002, expected schema
- **Dependencies:** T-002
- **Expected Output:** Validator that returns a list of structural errors (row number, field, error type, message) or an empty list on success.
- **Definition of Done:** Catches all structural edge cases defined in the spec. Returns row-level error detail. Unit tests pass.
- **Validation Notes:** Test every edge case from the spec table.

### T-004: Business Validator
- **Scope:** Implement validation for business rules: duplicate email detection (within file and against existing users), data type validation, field length constraints.
- **Inputs:** Parsed CSV data from T-002, database access for duplicate checking
- **Dependencies:** T-002
- **Expected Output:** Validator that returns a list of business rule errors or an empty list on success.
- **Definition of Done:** Detects duplicate emails within the file and against the database. Validates data types and field constraints. Unit tests pass.
- **Validation Notes:** Test duplicate detection (within file, against DB), type validation, constraint violations.

### T-005: Import Executor
- **Scope:** Implement the transactional import: open database transaction, batch-insert users (100 per batch), assign `import_batch_id`, commit or rollback.
- **Inputs:** Validated rows from T-003/T-004, database connection
- **Dependencies:** T-003, T-004
- **Expected Output:** Executor that imports users transactionally, returning success with count or failure with rollback confirmation.
- **Definition of Done:** Full transaction support — commit on success, rollback on failure. Batch insert works correctly. `import_batch_id` assigned to all imported users. Unit and integration tests pass.
- **Validation Notes:** Test successful import, rollback on error, batch insertion, `import_batch_id` assignment.

### T-006: Upload Endpoint and Error Report Generator
- **Scope:** Implement the `POST /api/users/import` endpoint with multipart file handling, auth checks, file size validation, and the error report generator that produces a CSV file and returns a download URL.
- **Inputs:** All core components from T-002–T-005, auth middleware
- **Dependencies:** T-005
- **Expected Output:** Working endpoint that handles the full upload-validate-import flow. Error report generator that creates CSV files and stores them temporarily.
- **Definition of Done:** Endpoint handles success and failure flows. Auth and file size checks work. Error report is generated, stored with 24h TTL, and downloadable. Integration tests pass.
- **Validation Notes:** Test full flow end-to-end. Test auth, file size limits, error report download, report expiry.
